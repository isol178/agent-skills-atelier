# deliberate — 設計相談の意思決定フレームワーク

この局面の成果物はコードではなく**構造化された判断材料と、確信度付きの推奨**である。Edit / Write でプロダクトコードを書き始めない。以下の型に沿って議論を進めるが、全ステップを機械的に踏む必要はない — 問いの重さに応じて省略する。**どこまで深掘りするかを決めるのがステップ2（可逆性）**なので、そこだけは飛ばさない。

## 1. 問いを再定式化する

ユーザーの発話から「実際に決めるべきこと」を1文で言い直し、冒頭で提示する。このとき：

- **複数の独立した決定が混ざっていれば分割する**。「通知をどう実装するか」は「チャネルは何か」「送信タイミングは何か」「失敗時リトライはどうするか」の3つかもしれない。まとめて議論すると全部が中途半端になる
- **一段上の問いが隠れていないか確認する**。「X をどう実装するか」の背後に「そもそも X は今のスコープに要るか」が隠れていることがある。隠れた問いを見つけたら先にそちらを提示する

## 2. 可逆性で分析の深さを決める

決定を two-way door / one-way door に分類し、投下する分析量を変える：

- **two-way door**（あとで安く戻せる：内部実装の構成、UIの細部、命名）→ 深掘りしない。推奨を1つ即答し、「これは可逆なので、走らせて違和感が出たら戻すのが最速」と添える。**過剰分析は可逆な決定に対する最大のアンチパターン**
- **one-way door**（戻すコストが高い：DBスキーマ、公開API契約、認証方式、データモデルの原則、外部サービス選定）→ 以降のステップを本気でやる

判断に迷う中間帯（マイグレーションで戻せるが痛い、等）は「戻すのに何が必要か」を具体的に書き出して分類する。

## 3. 制約と好みを分離する

評価に入る前に hard constraints を列挙する。出所の典型：

- プロジェクトの CLAUDE.md・設計原則・既存 ADR に明文化されたアーキテクチャ原則
- 予算・運用コストの上限、チーム規模と運用体制
- 互換性・コンプライアンス・セキュリティ要件

制約に反する案は「魅力的だが不可」として**評価対象から外す**。制約と好み（あったら嬉しい性質）を混ぜると、決定ドライバの比較が濁る。ユーザーが既に決めたことも制約として扱う（再議論しない）。

## 4. 選択肢を並べる（最低3案）

- **do-nothing / 現状維持を必ず含める**。それが基準線になり、他案の価値が測れる
- **「自然なデフォルト」案を必ず含める**。フレームワークの普通の使い方・業界標準。これを外す決定はまさに ADR に残すべき判断である
- 2案しか思いつかないときは、両案の共通前提を疑うと第3案が出ることが多い（例：「A社かB社か」→「そもそも外部サービスなしで済むか」）

## 5. 評価する

- 決定ドライバ（性能・コスト・運用負荷・拡張性・実装速度）が**衝突するときだけ**比較表にする。全案が一つのドライバで決まるなら表は不要、1段落で書く
- **二次効果を1つは書く**：「この決定が正しかったとして、6ヶ月後に何が起きているか」。例：DB拡張の採用自体は正しくても、マネージドDBの移行先候補が狭まっている、等
- 不確実な前提には「検証する最小の実験」を提案する（スパイク実装・料金試算・API 制限の実測）。議論で潰せない不確実性は議論を続けても潰れない

## 6. 推奨と反転条件をセットで出す

推奨は必ず1つに絞り、確信度（高/中/低）を添える。そして**反転条件（reversal trigger）を必ず書く**：

> 推奨：X（確信度：中）。ただし「週次バッチの実行時間が10分を超える」が観測されたらこの判断は見直すべき。

反転条件のない推奨は出さない。反転条件は将来の自分への「この決定を疑ってよいタイミング」の申し送りであり、ADR の Confirmation 欄の原料にもなる。

## 7. 出口

- **決定に至ったら**：コードを読むだけでは「なぜそうなっているか」を復元できない判断なら、ADR として記録することを提案する（プロジェクトに adr skill や `docs/decisions/` 等の流儀があればそれに従う）。実装に進むなら implementation-planning skill へ
- **未決で終わったら**：無理にまとめない。「残った open questions」と「それぞれを解消する最小アクション」を列挙して終わる。未決を未決のまま明示するのも正当な成果物

## アンチパターン

- 網羅的な選択肢サーベイの垂れ流し（推奨のない5案比較は判断材料ではなく宿題の転嫁）
- two-way door への過剰分析・one-way door への即答
- ユーザーが既に決めたことの再議論（決定済み事項は制約として扱う）
- 「実装しながら考えましょう」で議論を打ち切ること（それは implementation-planning の領分に入ってから）
