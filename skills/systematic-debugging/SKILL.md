# systematic-debugging — 系統的デバッグの思考プロトコル

デバッグの失敗パターンは decade を超えて同じだ：最初に思いついた仮説に飛びつき、確かめる前に直し、直らないので別の場所も直し、どの変更が効いたか分からなくなる。このプロトコルは「修正」を最後に置き、それまでを**観測と切り分け**に使う。

## 0. 既知の罠を先に照合する

チームで開発しているコードベースでは、「新奇に見えるバグ」のかなりの割合が既知の罠の再演である。プロジェクトに罠の記録（tips 文書・troubleshooting・skill 群・過去の postmortem・issue）があれば、深掘りを始める前に2分だけ使って照合する：

```bash
grep -ri "<症状のキーワード>" docs/tips/ .claude/skills/ 2>/dev/null
```

典型的な「既知の罠」の匂い：dev では動くのに本番ビルドだけ落ちる（ビルドモード差）、GET は通るのに POST だけ認証エラー（ミドルウェアの非対称）、依存を更新しても脆弱性警告が消えない（間接依存）。この種の症状は誰かが先に踏んでいることが多い。

## 1. 症状を「観測」と「推測」に分離して書く

調査の最初に、必ずこの2行を分けて書く：

- **observed**: 実際に観測した事実のみ。「POST /schedules が 500 を返す。レスポンスボディは {...}。サーバーログに IntegrityError」
- **inferred**: そこからの推測。「おそらく NULL が NOT NULL カラムに入っている」

推測を観測として扱った瞬間、調査は推測の上に推測を積む砂上の楼閣になる。「エラーが出ている」ではなく「**どこで**観測した**どの**エラーか」まで書く。エラーメッセージは**最後の行まで読み切る**（スタックトレースの1行目だけ読んで検索を始めない）。

## 2. 再現を固定化する

- 最小の再現手順（コマンド1つ・テスト1つ・curl 1発）に落とす。再現手順が3ステップを超えるうちは切り分けを始めない — 実験のたびに3ステップ踏むコストで思考が濁る
- 再現しないバグは「再現条件の特定」自体を最初の課題にする。直近で変わったもの（コミット、依存、環境変数、ブランチ）を `git log --oneline -10` / `git diff` で列挙する
- 「さっきまで動いていた」なら `git bisect` が最速の場合が多い。手動の推理より先に検討する

## 3. 仮説台帳をつくる

仮説を**3つ以上**列挙し、各仮説に「その仮説が真なら観測されるはずのこと」を書く：

```
H1: バリデーション通過後に NULL 書き込み → 真なら: 422 でなく 500、ログに IntegrityError ✓観測済み
H2: 認証ミドルウェアで user_id が取れていない → 真なら: 401 が返るはず ✗観測と矛盾
H3: migration 未適用でカラムが無い → 真なら: UndefinedColumn エラーのはず → 要確認
```

3つ書く理由：1つ目の仮説は「思いつきやすい」だけで「正しい」わけではない。台帳に書くと、**観測と既に矛盾している仮説**（H2）を無意識に追いかけるのを防げる。

次の実験は「最も識別力が高いもの」= 一度に最多の仮説を切り分けられるものを選ぶ。仮説を確認する実験より、**仮説を棄却できる実験**の方が価値が高い。

## 4. 層で二分探索する

Webアプリの典型的なリクエスト経路は層が深い：

```
client (UI / 状態管理) → API境界 (fetch / 認証ヘッダ) → ルーター
→ サービス → データアクセス → DB
                 └→ 外部サービス（LLM・決済・メール等）
```

端から順に追わず、**真ん中の境界で観測して半分に切る**。例：フロントで500が見えるなら、まず同じリクエストを curl（またはAPIテスト）でサーバーに直接投げる。再現すればクライアント層は無罪、しなければリクエストの組み立てが被疑。外部サービス絡みなら、トレース・リクエストログでどの呼び出し単位まで正常かを切る。

## 5. 修正は1度に1変更

- 修正候補を2つ同時に入れない。どちらが効いたか永久に分からなくなり、片方は不要な変更としてコードベースに残る
- 「たぶんキャッシュ」「とりあえず再起動」で直った場合は、**直っていない**（再発条件を特定できていない）。時間の制約でそこで止めるなら、その旨を明示して記録に残す

## 6. 停止条件と反省

- **実験3回で全仮説が棄却されたら、前提を疑う**：正しいブランチ・環境・DB・バージョンを見ているか。「見ているファイルが実行されているファイルか」（ビルドキャッシュ、別プロセス、stale な依存）。バグ調査が長引く原因の上位は「そもそも観測対象が間違っていた」
- 直ったら2つを残す：(1) 回帰テスト、(2) 「なぜ最初から見えなかったか」の1行 → 非自明なら、プロジェクトの知見置き場（tips・skill候補リスト等）へ記録する

## アンチパターン

- **shotgun debugging**：印象で複数箇所を書き換えて祈る
- エラーメッセージを読み切る前に検索・修正を始める
- 「再現したりしなかったりする」を許容したまま修正に進む（並行性・環境差・キャッシュの3択であることが多い）
- 調査ログを残さず、同じ実験を2回やる
