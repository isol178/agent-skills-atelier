# maintaining-docs-for-jit

設計書（Markdown）を節単位ファイルに分割してJIT参照を可能にする。
分割済みディレクトリを1ファイルに結合して全体レビューやLLM渡しにも使える。

## スクリプトについて

`scripts/` 以下のスクリプトは信頼できる実装である。中身を読む必要はない。
呼び出しコマンドと出力の解釈のみを行うこと。

| スクリプト | 役割 |
|---|---|
| `scripts/estimate.py` | 分割シミュレーション・品質チェック・パターンベース独立性リスク検出 |
| `scripts/split.py` | 実際の分割実行（ファイル生成） |
| `scripts/merge.py` | 分割ディレクトリを1ファイルに結合 |

---

## 操作モード

ユーザーが明示しない場合は必ずモードを確認してから進む。

| モード | 概要 |
|---|---|
| `split` | 1つのMDファイル → 節単位ファイル群 + `INDEX.md` |
| `merge` | 分割済みディレクトリ → 1つの結合ファイル（元ファイルは変更しない） |

---

## split モード

### Step 1: 引数収集

ユーザーから確認する情報：

| 項目 | 必須 | デフォルト |
|---|---|---|
| 対象ファイルパス | yes | - |
| 分割見出しレベル | no | Step 3 で決める |

### Step 2: 全文精読と意味的独立性の判断

**分割前にファイルを全文読む。** これはスクリプトでは代替できない。

読みながら以下を判断する：

**意味的独立性チェック**

各節が単独で意味をなすかを確認する。以下のパターンは独立性を損なう：

- ある節の内容が、別の節で定義された概念を前提として説明されている
- 節内の表・リストの項目が、他節の内容を参照しないと意味が通じない
- 「上記の設計方針に基づき〜」のように直前節の結論を受けて話が展開している
- 「詳細は §N を参照」のような外部参照（INDEX.mdで補完可能だが要注意）

**構成変更の提案**

独立性を損なう箇所が複数ある場合、そのまま分割するより先に構成変更を提案する。
例：「§3 と §4 は密結合しているため、まとめて1節にするか、共通の前提を §2 末尾に移動することを推奨します」

ユーザーが構成変更を希望する場合はここで対応してから Step 3 へ進む。
「リスクを承知で分割」を選んだ場合もそのまま Step 3 へ進む。

### Step 3: 見積もり実行

全レベル比較で俯瞰する：

```bash
python scripts/estimate.py <ファイルパス> --all-levels
```

候補レベルが絞れたら詳細見積もりを実行する：

```bash
python scripts/estimate.py <ファイルパス> --level <N>
```

### Step 4: 見積もり結果の解釈とユーザー確認

estimate.py の出力を読み取り、ユーザーに要約して伝える。

**品質基準**

| 基準 | 判定 | 対応 |
|---|---|---|
| 節数 3〜25 | OK | 続行 |
| 節数 1〜2 | WARN | より細かいレベルを提案 |
| 節数 26以上 | WARN | より粗いレベルを提案 |
| 10行未満の節が全体の50%以上 | WARN | レベルを1段上げることを提案 |
| 300行超の節が存在 | INFO | 該当節名をユーザーに通知。続行可 |

出力行の `[WARN]` が1件以上あれば問題あり。別レベルを提案する。
`[INFO]` は注意喚起のみ。続行可。すべて `[OK]` なら続行推奨。

**パターンベース独立性リスク（estimate.py の検出）**

「節をまたぐ可能性のある参照」セクションを確認する。
検出なし なら理想的。件数がある場合はユーザーに報告し、以下を委ねる：
- 参照元の表現を修正してから分割（推奨）
- リスクを承知で分割（INDEX.md に注記を追加）
- 分割レベルを変更して参照が節内に収まるようにする

ユーザーの確認が取れたら Step 5 へ進む。

### Step 5: split 実行

```bash
python scripts/split.py <ファイルパス> --level <N>
```

### Step 6: 分割後の確認

split 完了後、ユーザーに伝える：

1. 生成ファイル一覧（出力ディレクトリ名と節数）
2. バックアップ（`<元ファイル>.md.bak`）が作成された旨
3. Step 2・Step 4 で検出した独立性リスクを再掲し、該当節ファイルの確認・修正を促す

## merge モード

### Step 1: 引数収集

対象ディレクトリパスを確認する。

### Step 2: merge 実行

```bash
python scripts/merge.py <ディレクトリパス>
```

### Step 3: 結果確認

- 出力ファイルパス（`<ディレクトリ名>_merged.md`）を伝える
- 用途（LLM渡し・印刷・全体レビュー）に応じてそのまま使うよう案内する
- 元の分割ファイルは変更されていないことを明示する
