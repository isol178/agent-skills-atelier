# テンプレート：フル計画書

複数フェーズ・委譲前提・長期間（セッションを跨ぐ）の作業向け。リファクタ・移行のように
**「変えないこと」（不変条件）が成果物の本質**である作業は、規模が小さくてもこちらを使う。

## このテンプレートの使い方

- 骨格をコピーし、`<>` を埋める。**該当しない節は削除する（空欄・「N/A」で残さない）**。
  空節は「書くことを検討して不要と判断した」のか「書き忘れた」のか読者に区別できない
- 各節の `> 注釈：` は執筆ガイドなので、書き終えたら削除する
- 課題・決定・タスクには**ID を振る**（課題 K1..、決定 D1..、フェーズ P1-1.. 等。接頭辞は計画ごとに自由）。
  ID があると PR・コミット・引き継ぎメモから計画の該当行へ一意に参照でき、「あの件」問題が消える
- 調査に基づく記述には**実測日を書く**。計画は書いた瞬間から古くなり始める。日付がないと
  「今も正しいか再確認すべきか」を後続セッションが判断できない

---

## 骨格

```markdown
# <計画名>（<一言で内容がわかる副題>）

**文書バージョン：v1.0**
**作成日：YYYY-MM-DD**
**更新日：YYYY-MM-DD**
**ステータス：<未着手 / P1 完了・P2 着手中 等。進行のたびに必ず更新>**

> <この計画が何の調査に基づくか（日付）。各タスクは対象ファイル・不変条件・検証コマンド・
> 完了基準まで固定してあること。進行に合わせて進捗チェックリストと引き継ぎメモを更新する義務>

## 0. この文書の使い方（実施プロトコル）

> 注釈：後続セッション（別インスタンス・サブエージェント含む）が本文を読む前に従うべき
> 運用ルール。ここが計画の「実行可能性」を担保する。最低限：

1. **フェーズ順序と依存関係**：<P1 → P2 → …。並行着手可の組み合わせと、着手条件
   （「P3 は P1 完了 + 外部入力 X の確定が前提」等）を明記>
2. **1タスク = 1 PR** を基本とする。<例外があれば理由付きで明記>
3. **委譲の作法**：サブエージェント委譲時は委譲側プロトコル（読むべき skill の明示等）に従い、
   本文書の該当タスク節を**そのまま**契約として渡す
4. **全タスク共通の不変条件**：<「描画結果を変えない」「API 契約に触れない」「文言に触れない」等。
   違反に気づいたら作業を止めて報告する>
5. <プロジェクト所定の出荷前チェック・タスク終了ルーチンがあればここに>

## 1. 背景・目的

> 注釈：なぜやるか、完了後に何がどう変わるかを数段落で。設計思想（例：「①配管の一本化 →
> ②整合修繕 → ③値の刷新、に分離する」）があればここに書く。核心の問いに答える形
> （「Xだけで済むか？→済まない、なぜなら…」）が有効なこともある

## 2. スコープ境界

> 注釈：対象・対象外を表で裁定する。**対象外にこそ理由を書く**（「別計画の領分」「技術制約」等）。
> 境界事例（同じ要素が対象と対象外の両方に現れるケース）は脚注で明示的に潰す。
> 境界が自明な計画ではこの節は「スコープ外（明示）」の箇条書きに縮めてよい

## 3. 調査結果（YYYY-MM-DD 実測）

> 注釈：計画の根拠となる棚卸し・課題一覧。再現可能にするため**検索コマンドや確認手順ごと**書く。
> 課題には ID（K1..）を振り、「該当箇所」と「どのフェーズで対応するか」を列に持つ表にする。
> 調査対象が単純なら §1 に吸収して削除してよい

## 4. 確定設計（実装前に固定する決定事項）

> 注釈：着手後に議論を再燃させないための決定の固定。決定には ID（D1..）を振る。
> 「何を採用したか」だけでなく「何を不採用にしたか・なぜ」も1行ずつ。
> コードから復元できない判断は ADR に切り出し、ここには結論と ADR 参照だけを残す

## 5. フェーズ分解

> 注釈：フェーズごとに1節。各タスクは「そのまま委譲契約として渡せる」粒度まで固める。
> タスクの型は固定：

### P1. <フェーズ名>（<逐次/並列可・委譲可否>）

**対象ファイル**：<新規・変更の全列挙。これが委譲時のファイル所有権契約になる>

1. <手順。パターンアンカーにすべき既存ファイルがあれば名指しする>
2. <手順>

**不変条件**：<このフェーズで変えてはならないもの>
**検証コマンド**：<コピペで実行できる形で>
**完了基準**：<検証がどうなったら完了か。「動いたら」は不可>

## 6. リスク（premortem）と対策

> 注釈：「手戻りになったとして原因は何だったか」を3件程度。各リスクに
> 「計画のどこで潰しているか」（§参照）を紐付ける。潰せないリスクは検証順序を前倒しする

## 7. 検証ゲート（全フェーズ共通）

> 注釈：レイヤー×コマンドの表。CI と同一の invocation を書く（ローカルだけ通る書き方をしない）

## 8. 進捗チェックリスト

> 注釈：フェーズ×タスクのチェックボックス。完了時は日付・コミット/PR 参照を添える。
> **引き継ぎはここから読まれる**前提で維持する

## 9. 引き継ぎメモ

> 注釈：調査時の既知事実（後続が再調査しなくて済む知見）・他計画/他作業との接点
> （同一ファイルを触る計画、直列にすべき PR 等）・未決事項とその解消条件。
> 未決事項は解消したら「裁定」として残す（消さない — 判断の履歴が価値）

### 変更履歴

| 日付       | 版   | 変更 |
| ---------- | ---- | ---- |
| YYYY-MM-DD | v1.0 | 初版 |
```
