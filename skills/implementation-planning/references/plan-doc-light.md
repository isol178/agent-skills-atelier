# テンプレート：軽量実装計画

単機能・単フェーズだが、セッションや担当を跨ぐ可能性がある実装向け。
フル計画書の運用規律（実施プロトコル・premortem 節・検証ゲート表）を省き、
「決定の固定」と「引き継ぎ可能性」だけを残した形。

## このテンプレートの使い方

- **該当しない節は削除する（空欄で残さない）**。`> 注釈：` は執筆ガイドなので書き終えたら削除する
- 途中でフェーズが2つ以上に割れた・委譲が必要になった・不変条件の管理が本質になってきたら、
  軽量のまま肥大させず**フル計画書（plan-doc-full.md）の骨格に移行する**
- 1PR・1セッションで完結する見込みなら文書自体が不要。SKILL.md §5 のインライン提示で足りる

---

## 骨格

```markdown
# <機能名> 実装計画

**作成日：YYYY-MM-DD / ステータス：<未着手 / 実装中 / 完了>**

## 1. ゴールとスコープ

> 注釈：何ができたら完成か（1〜3行）と、やらないことの明示。
> 要件ID・チケットID・参照すべき設計文書の節をここで列挙する

## 2. 設計上の決定事項（実装前に固定）

> 注釈：着手後にブレると手戻りになる判断を先に固定する。採用だけでなく
> 「見送ったこと・理由」も1行ずつ。コードから復元できない判断は ADR へ切り出す

## 3. 追加・変更ファイル一覧（作業分解）

> 注釈：ファイルパス × 何をするか × 依存順序。パターンアンカーにする既存ファイルがあれば
> 名指しする。委譲するならここがファイル所有権の契約になる

| ファイル | 変更内容 | 備考（依存・アンカー） |
| -------- | -------- | ---------------------- |

## 4. 進捗チェックリスト（引き継ぎ時はここを見る）

> 注釈：作業分解と1対1のチェックボックス。完了時に日付・コミット参照を添える

## 5. 検証手順

> 注釈：コピペで実行できるコマンドと、手動確認の操作手順（どの画面で何をして何が見えるか）。
> CI と同一の invocation を書く

## 6. 引き継ぎメモ（随時更新）

> 注釈：調査済みの既知事実・ハマった点・未決事項。空なら節ごと削除
```
