# implementation-planning — 実装着手前の計画フレームワーク

まとまった実装の手戻りの主因は「コードが書けないこと」ではなく、**正典とズレたまま書き始めること**と**既知の罠を再演すること**である。着手前に以下を通し、短い計画を提示してから書き始める。

## 1. スコープと正典を確定する

### スコープの出所を辿る

開発計画（マイルストーン・チケット）→ 要件定義 → 該当する画面仕様・API設計・DBスキーマ文書、の順で読む。設計書が節ごとに分割されていれば該当節のみ読む（全節の一括読み込みはしない）。要件ID / チケットIDのリストを計画に明記する — 「なんとなくこの画面」ではなく識別子で書く。

### 「文書より実装が正」の箇所を先に特定する

たいていのプロジェクトには文書が実装に追いついていない箇所が存在する。書き始める前に突き合わせる。典型的な要注意箇所：

| 対象 | 正典になりやすいもの |
|---|---|
| DBカラム名・型 | migration ファイル（設計書は遅れがち） |
| enum・定数の有効値 | 実装のバリデーション定義（文書は表示名のみのことがある） |
| レスポンス形式（エンベロープ・キー名） | 既存エンドポイントの実装 |
| 状態遷移 | プロジェクトごとに異なる — どちらが正か先に確認する |

プロジェクトの CLAUDE.md や規約に正典の対応表があればそれに従う。乖離を見つけたら**実装前に報告**し、文書修正を同一PRのスコープに入れる（「あとで直す」はしない）。

## 2. premortem — 「手戻りになったとして、原因は何だったか」

着手前に、**この実装が手戻りになった未来を仮定して、その原因を3つ書く**。事前に「うまくいくか？」と自問しても楽観が答えるだけだが、「失敗した。なぜか？」と問うと具体的なリスクが出てくる。

プロジェクトに既知の罠の記録（tips 文書・troubleshooting・skill 群・過去の postmortem）があれば、必ず照合する：

```bash
grep -ri "<関連キーワード>" docs/tips/ .claude/skills/ 2>/dev/null
```

premortem で挙げたリスクのうち、**設計で潰せるものは計画に反映し、潰せないものは「最初に検証する」順序に**する（リスクの高いものから書く — 最後に発覚すると手戻りが最大になるため）。

## 3. 逐次と並列に分解する

作業を2種類に分ける：

- **逐次必須（メインセッションで先にやる）**：DBスキーマ・migration、共有コンポーネント・共有型定義、API契約の確定、設計判断すべて
- **並列可能（委譲できる）**：確定した契約の上に乗る独立した画面・ルート・テスト

設計判断を含む作業は委譲しない。**設計を計画文書（またはプロンプト内の契約）に固定してから**、単純化された作業のみをサブエージェントに渡す。委譲プロンプトには読むべき skill を明示的に列挙する（最低 `delegated-task-protocol`、新規ファイル作成を含むなら `pattern-anchoring` も）— サブエージェントの skill 自動発動は当てにしない。複数エージェントを並列に走らせるなら最低限：ファイル所有権を重複させない・共有コンポーネント/共有型は先にメインで作る・マージ後に全体の型チェックを回す（プロジェクトに並列運用の規約があればそちらを優先）。

分解の健全性チェック：各並列タスクのプロンプトが「参照すべき契約」を自己完結で含んでいるか。エージェントに推測させる箇所が残っていたら、それは分解がまだ粗い。

## 4. 完了基準を先に書く

「動いたら完了」は完了基準ではない。着手前に列挙する：

- [ ] どのテスト（既存・新規）が通ったら機能完了か
- [ ] 手動確認の手順（どの画面で何を操作して何が見えるか）
- [ ] 出荷前チェック（lint / format / test — プロジェクト所定のコマンドがあればそれ）の通過
- [ ] ドキュメント波及の更新（spec → API設計 → DBスキーマ → 画面仕様。**同一PR**に含める）
- [ ] コードから復元できない設計判断をしたか → したなら ADR に記録
- [ ] プロジェクトのタスク終了ルーチン（知見の記録・ナレッジグラフ更新等）があれば実施

## 5. 計画の提示フォーマット

書き始める前に、以下を簡潔に提示する（長文にしない。各項目2〜5行）：

```markdown
## 実装計画：<対象>
- **スコープ**: 要件ID / チケットID / 対象ファイル群
- **正典確認**: 突き合わせた結果（乖離があれば明記）
- **リスク（premortem）**: 3件と、各対策
- **分解**: 逐次パート / 並列パート（委譲する場合は契約の所在）
- **完了基準**: 上のチェックリストの具体化
```

ユーザーの依頼が明確な実装指示であれば、提示後すぐ着手してよい（承認待ちで止まらない）。方針レベルの不確定要素が残っている場合のみ、その1点に絞って確認する。

## 6. 計画を文書として残す場合

計画がその場のインライン提示（§5）で消費しきれない規模なら、計画書ファイルとして残す（置き場所はプロジェクトの流儀に従う。例：`docs/plans/`）。文書化された計画は後続セッション・サブエージェントへの**契約**として機能する — だからこそ形式を揃える価値がある。規模に応じてテンプレートを選ぶ：

| 条件（いずれかに該当） | 使うもの |
|---|---|
| フェーズが2つ以上 / サブエージェント委譲の予定がある / 「変えないこと」（不変条件）の管理が本質のリファクタ・移行 | `references/plan-doc-full.md` |
| 単機能・単フェーズだが、セッションや担当を跨ぐ可能性がある | `references/plan-doc-light.md` |
| 1PR・1セッションで完結する見込み | 文書化しない。§5 のインライン提示のみ |

迷ったら軽量から始めてよい — 軽量テンプレートには「フルへ移行すべきシグナル」が書いてある。逆にフルで書き始めて節が埋まらないのは、軽量で十分だったサイン。どちらのテンプレートも「該当しない節は削除する（空欄で残さない）」を原則とする。
