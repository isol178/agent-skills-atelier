# 議論ルール

> Phase 5 Deliberate と Phase 6 Synthesize の実行時ルール集。
> ここにあるのは**実行中に従う指示**のみ。
> ルールの背景・設計思想は README.md を参照。

## 同意しすぎ(Sycophancy)対策

Phase 5-6 全体を通じて以下を守る:

- 対立ロールは構造に組み込む(DA は必ず1名、ペルソナで明示する)
- 独立意見生成フェーズ(Phase 4)では他エージェントの意見を一切見せない
- 多数派情報をエージェントに渡さない(「3人中2人が賛成」等は禁止)
- フル議事録ではなく対立点の要約差分のみを渡す

## 少数意見の保護

### 必須ルール(全深度)

1. Phase 5 で反論を求める際、多数派情報は伝えない(「別の見解がある」とだけ)
2. Phase 6 で少数意見チェックを**省略しない**
3. 少数意見を棄却する場合、**具体的な理由を必須記述**する
4. 棄却理由が不十分なら、少数意見をレポートに含める(疑わしきは採用)

### 少数意見チェックの手順

SKILL.md Phase 6 の 2x2 マトリクス(根拠有無 × 影響大小)が判定基準の正。
以下はその実施手順:

```
1. 全エージェント出力から多数派と異なる意見を抽出
2. 各少数意見について順に判定:
   a. 具体的な根拠があるか? なければ棄却可(理由: 根拠なし)
   b. 他エージェントが具体的に反論しているか?
      - 反論が具体的 → 棄却可(理由: [反論の要約])
      - 反論が抽象的 → 採用
   c. この意見が正しかった場合の影響は?
      - 影響大 → 採用(論理的な懸念があり、リスクが高い場合は根拠が限定的でも採用する)
3. 採用された少数意見は「少数意見」セクションに記載
4. 棄却された少数意見は棄却理由とともに記録(Deep のみ)
```

## 収束判定

Phase 5 は以下のいずれかを満たせば収束と判定する:

1. **全員合意**: 全エージェントが同じ結論に達した
2. **新規論点なし**: Round N の出力に、Round N-1 にない**具体的証拠・データ・事例**を含む論点が含まれていない(言い換えは新規論点ではない)
3. **ハードキャップ**: Round 2 に到達

Quick 深度では Phase 5 をスキップ。対立があっても議論せず、
Phase 6 の重み付けで解決する。

## 重み付け基準(Phase 6)

自己申告の確信度は使わない。以下の基準で重み付ける:

### 基準1: 根拠の具体性

| レベル | 例 | 重み |
|---|---|---|
| 高 | コード例、データ、仕様書引用を含む | 最重視 |
| 中 | 具体的な事例や経験に基づく | 標準 |
| 低 | 「一般的に」「ベストプラクティスとして」等の抽象的主張 | 低い |

### 基準2: 専門領域との一致度

論点に最も関連する専門家の意見を重視する。
例: セキュリティの論点 → セキュリティ専門家の意見が最も重い。

### 基準3: 反論応答の品質

反論に具体的に応答できた意見を重視する。
反論を無視した、または「それでも私の意見は正しい」のみで応答した意見は軽視する。

## オーケストレーターの行動規範

### Phase 4-5: プロセス管理者モード

**やる**:
- 議題と評価基準を明確にする
- 各エージェントに必要な情報のみを渡す
- 対立点を正確に要約する(下記「要約品質ルール」に従う)
- 少数意見を保護する
- DA の批判品質を管理する(下記「DA 品質管理ルール」)

**やらない**:
- 自分の意見を議論に入れる
- 多数派情報をエージェントに渡す
- DA に合意傾向や多数派の方向性を伝える
- エージェントに「正しい答えを出せ」と指示する
- 3ラウンド以上の議論を行う

### Phase 6: 統合者モード

**やる**:
- 重み付け基準(上記3基準)を機械的に適用する
- 少数意見チェックを省略しない
- 判断理由を明示する(「なぜこちらを採用したか」を必ず記述)

**やらない**:
- 基準にない独自の判断軸を持ち込む
- 少数意見チェックを省略する
- エージェントの自己申告確信度を採用する

## 要約品質ルール(Phase 5)

対立点の要約はこのスキルのシングルポイントオブフェイラー。
以下を守る:

- 元の主張の**結論**と**根拠**を両方含める(結論だけの要約は不可)
- 具体的なデータ・事例の引用は省略しない
- 要約によって一方の主張が有利/不利に見えるフレーミングをしない

**Deep 追加ルール**:
要約とともに元の出力の該当箇所を引用として渡す。
さらに要約を伝える前に各エージェントに対称性チェックを行う:
「この要約はあなたの主張を正確に反映しているか、修正点はあるか」

**Standard 代替ルール**(対称性チェックは行わないが):
要約と併せて元の出力の該当箇所を引用として渡す(要約のみへの依存を緩和)。

## DA(Devil's Advocate)品質管理ルール

DA の批判品質はオーケストレーターが管理する。ペルソナ自己規律だけに依存しない。

### 過剰批判の判定

以下のいずれかに該当したら批判が過剰または低品質:

- 具体的な根拠や失敗シナリオを伴わない抽象的批判が3つ以上
- 他の全専門家の全発言に対して反対している(焦点が絞れていない)
- 同じ批判を異なる言い回しで繰り返している

**介入**:
- Phase 5 で検知したら再指示する:
  「最も構造的な弱点1つに絞って、具体的な失敗シナリオとともに再分析せよ」
- Phase 6 の重み付けでは、根拠のない批判を明示的に低く扱う

### 批判不十分の判定

以下のいずれかに該当したら批判が形骸化:

- 他の専門家の発言の言い換えにとどまり独自視点がない
- 「潜在的なリスクがある」「考慮すべき点がある」等の指摘止まりで
  具体的な失敗シナリオがない
- Quick 深度で DA の批判が全専門家の合意点と矛盾しない(批判として機能していない兆候)

**介入**:
再指示する:
「もしこの提案/対象が最悪の形で失敗するとしたら、
その原因を1つの具体的なシナリオで描写せよ」

## 対立点の伝え方(Phase 5)

### 良い伝え方

```
あなたの分析では「モバイルファーストのレイアウトが適切」としていますが、
別の専門家は「デスクトップユーザーが70%を占めるためデスクトップ優先が
適切」と分析しています。

この対立について、あなたの観点から改めて分析してください。
意見を変える場合も維持する場合も、具体的な根拠を示してください。
```

### 悪い伝え方

```
他の3人の専門家はデスクトップ優先に賛成しています。
あなたの意見を再考してください。
```

→ 多数派圧力で少数意見が消える。使用禁止。

## セカンドチャンス判定ルール(Phase 6 Deep のみ)

意見変更の判定は**差分の有無ではなく変更理由の質**で行う:

| 変更理由 | 判定 |
|---|---|
| 新たな根拠(データ、事例、論理的推論)がある | 正当な変更 → 採用 |
| 「統合結果を見て納得した」レベル | sycophancy の兆候 → 元の意見を尊重 |
| Step 1 で「変わらない」と答えたのに Step 2 で根拠なく変更 | sycophancy → 元の意見を尊重 |

正当な変更と判定された指摘のみ最終レポートに反映する。
