# ハーネスアダプタ

> Phase 4 の並列独立実行、モデル混合、Phase 5 の直接対話オプション、
> および並列実行不可時の fallback を環境別にまとめる。
>
> SKILL.md 本体はハーネス非依存の抽象概念のみで書かれている。
> ここで具体的な実装方法を環境ごとに隔離する。

## 並列独立実行の方式

Phase 4 は**並列独立実行**が必須。各エージェントが独立したコンテキストを持ち、
他の専門家の存在を知らない状態で分析する構造を作る。

### 環境別の起動方法

| 環境 | 起動方法 |
|---|---|
| Claude Code | Task tool(`subagent_type` を指定)を並列に呼び出す。必要に応じて `run_in_background: true` を使用 |
| Cursor | サブエージェント機能を使用し、専門家ごとに独立して並列起動する |
| Codex | `.agents/skills` 配下から専門家ごとのサブスキルを並列起動、または OpenAI API で並列リクエスト |
| OpenCode | サブエージェント機能を使用する |
| その他の API 直呼び環境 | 専門家ごとに独立した API リクエストを並列送信(同一会話履歴を共有しない) |

**共通要件**:
- 各エージェントは独立したコンテキストで起動する(会話履歴を共有しない)
- 起動時に渡すのは「ペルソナ + 議題 + 対象物 + 評価基準」のみ
- 他の専門家の存在・分析・意見を一切伝えない

### 結果の回収

全エージェントの分析出力を個別に保存し、全員分が揃ってから Phase 5 または Phase 6 に進む。
途中で1名でも欠落した場合はリトライする(欠落のまま統合しない)。

## モデル混合(任意)

異なるプロバイダーのモデルを専門家ごとに割り当てることで、
同一モデルの擬似的多様性問題を緩和できる。

### 推奨される混合パターン

**クロスプロバイダー混合**(推奨):
- 例: 専門家A = Claude、専門家B = GPT、専門家C = Gemini、DA = Claude
- 異なる学習データ・異なるアーキテクチャによる本質的な多様性が得られる

**同一ファミリー内での混合**(非推奨):
- 例: 上位モデル + 下位モデルの組み合わせ
- 多様性ではなく性能差が生じ、特定エージェントだけが「賢く」なって
  公平な議論が成立しない

### 環境別の設定方法

| 環境 | 設定方法 |
|---|---|
| Cursor | サブエージェントの `model` フィールドで異なるプロバイダーを指定 |
| Claude Code | Task tool では現時点で Claude モデルのみ。異プロバイダー混合には他ツール併用が必要 |
| API 直呼び | 専門家ごとに異なるプロバイダーのエンドポイントを呼び分ける |

### モデル混合が不可能な場合

同一モデルで実行してもスキルは正常に動作する。
ただしその旨をレポートの「議論の限界」に記載する:

> この分析は単一の LLM モデルによるペルソナベースの議論であり、
> モデル固有の盲点は検出できない。

## 直接対話オプション(Phase 5)

Phase 5 Round 1 では**オーケストレーター仲介方式がデフォルト**。
オーケストレーターが対立点を要約し、各エージェントから個別に反論を回収する。

環境が直接対話機能に対応している場合、特定の条件下で2名の専門家を
直接対話させることができる。

### 使用条件

以下を全て満たす場合のみ使用する:
- 2名の専門家が特定の論点で**具体的な根拠を示しつつ**対立している
- オーケストレーターの要約では論点の核心を捉えきれない懸念がある
- 環境が多エージェント直接対話に対応している

### 環境別の実装

| 環境 | 実装方法 |
|---|---|
| Claude Code (Agent Teams 対応版) | SendMessage で2名のサブエージェント間の対話を仲介 |
| Cursor (対応時) | サブエージェント間メッセージング機能を使用 |
| 非対応環境 | **使用しない**。オーケストレーター仲介のみ |

### 運用ルール

- 直接対話は各対立点につき**往復2回まで**(Round 1 + Round 2 の枠内に収める)
- オーケストレーターは対話内容を監視する
- **sycophancy(一方が相手に同調して意見を変える兆候)** を検知したら介入する
  - 介入例: 「あなたは Phase 4 で X と主張していましたが、今の同意は
    新たな根拠に基づくものですか、それとも相手の主張に納得しただけですか」

### 注意

直接対話機能は環境によっては実験的・非標準機能である。
環境が未対応なら**使わなくてもよい**(スキル全体の動作に影響しない)。

## Fallback: 並列独立実行が不可能な場合

### 事前確認

Phase 4 開始前に、環境が並列独立実行に対応しているか判定する。
対応していない場合、**ユーザーに通知し続行の確認を取る**:

```
この環境ではサブエージェントの並列実行が利用できないため、
逐次実行で代替します。独立性の保証が限定的になり、品質が低下する
可能性があります。以下から選択してください:

1. このまま逐次実行で続行
2. Quick 深度に変更して実行(議論フェーズを省略)
3. 中止
```

### 逐次実行のルール

ユーザーが続行を選んだ場合、以下を守る:

1. 専門家 A の分析を生成した後、専門家 B の分析時には
   A の出力を参照させない(コンテキストに含めない)
2. 各専門家の出力は個別に記録し、全員分が揃ってから Phase 5/6 に進む
3. DA は必ず最後に実行する(他の専門家の「合意傾向」をモデルが
   先読みしてしまう危険を避けるため、DA への入力からは
   他専門家の出力を完全に除外する)

### 限界の明示

同一コンテキスト内の逐次実行では、前の出力が物理的に残るため
独立性は完全には保証されない。以下をレポートの「議論の限界」セクションに記載する:

> 本分析は並列独立実行が利用できない環境で逐次実行された。
> 構造的な独立性の保証は限定的であり、並列実行と比べて
> 品質が低下している可能性がある。

---

## 環境判定チェックリスト

Phase 4 開始時に以下を順に確認する:

1. サブエージェント/並列 API 呼び出しが可能か → 不可なら fallback へ
2. モデル混合が可能か → 可能なら混合を提案、不可なら単一モデルで続行
3. 直接対話機能が利用可能か → 利用可能なら Phase 5 で対立点があれば活用を検討

判定結果はレポートの「議論プロセス」セクションに記載する(透明性のため)。
