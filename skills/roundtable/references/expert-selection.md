# 専門家選定

> Phase 2 Calibrate での専門家パネル構成のための参照資料。
> 原理セクションとアーキタイプ早見表から構成される。

## Part 1: 選定原理の詳細

SKILL.md 本体で示した5つの原理の詳細版。Standard 以上では必ず参照する。

### 原理1: 失敗検知

**「この議題で失敗したとき、誰が最初に気づくか?」**

その視点を持つ専門家がパネルに必要:

- Web サイトの表示崩れ → フロントエンドエンジニア
- ユーザー離脱率の悪化 → UX リサーチャー
- 法的リスク → 法務専門家
- 予算超過 → ファイナンシャルアナリスト
- セキュリティ侵害 → セキュリティエンジニア

### 原理2: 受け手の視点

**「この議題の受け手・影響を受ける人は誰か?」**

作り手だけでなく、受け手(ユーザー、読者、利用者)の視点を含める:

- 記事レビュー → ターゲット読者(エンジニア、デザイナー等)
- API 設計 → API 利用者(他チーム、外部開発者)
- 事業戦略 → エンドユーザー代弁者
- UI デザイン → アクセシビリティを必要とするユーザー
- 教材設計 → 実際の学習者

### 原理3: 盲点発見

**「この議題で見落とされがちな観点は何か?」**

直接関連しないが、重要な視点を持つ専門家を含める:

- EC サイト評価 → アクセシビリティ専門家(法的義務になりつつある)
- 技術アーキテクチャ → ビジネスアナリスト(技術的に正しくても事業に合わない場合)
- 製品ローンチ → カスタマーサポート(サポートしきれない仕様)

### 原理4: 隠れた専門知識

**「この議題で、一般的には招集されないが実は重要な知見を持つ分野はあるか?」**

原理1-3 で専門家を選んだ後、この問いを明示的に自問する。

既知の例(これに限らない):

- AI 開発 → 倫理哲学者
- プロダクト設計 → 行動経済学者(意思決定バイアス、ナッジ)
- 組織設計 → 生態学者(複雑適応系、レジリエンス)
- UX 設計 → 認知心理学者(認知負荷、注意の科学)
- コンテンツ戦略 → 言語学者(フレーミング、語用論)
- セキュリティ設計 → 犯罪学者(攻撃者の動機モデル)
- データベース設計 → 会計士(監査可能性の要件)

**重要**: 候補が浮かばなければ無理に入れない。
「入れるべき」とロジックで判断した場合のみ適用する。

### 原理5: 対立構造

**Devil's Advocate を必ず1名含める(専門家枠とは別枠)。**

DA はドメイン専門家ではなく批判の専門家。
議題によらず常に1名、批判の焦点のみ議題に応じて変える。

## Part 2: 選定チェックリスト

Phase 2 でパネルを確定する前に以下を確認する:

- [ ] 同じ側の専門家が2名以上いないか(例: 編集者 + コンテンツストラテジストは両方「作り手」)
- [ ] 受け手の視点が含まれているか(全員が「作る側」「評価する側」になっていないか)
- [ ] DA を専門家枠に数えていないか(DA は別枠で常に1名)
- [ ] パネルサイズは議題の複雑さに見合っているか(最小4名、最大8名)
- [ ] 原理4 を明示的に問うたか(隠れた専門知識の自問を省略していないか)

### よくある落とし穴

| 落とし穴 | 修正 |
|---|---|
| 編集者 + コンテンツストラテジスト(作り手 x 2) | 編集者 + ターゲット読者(作り手 + 受け手) |
| エンジニア3名(実装視点に偏り) | エンジニア + UX + PM(実装 + 体験 + 事業) |
| DA を専門家枠にカウント | DA は常に別枠で+1 |
| 全員が同プロバイダーからの専門家 | 失敗検知・受け手・盲点・隠れた知見のバランス |

## Part 3: Devil's Advocate の焦点設定

議題の性質に応じて DA が批判する対象を変える:

| 議題の性質 | DA の焦点 |
|---|---|
| 新しい提案 | 「なぜこれは失敗するか」 |
| 既存のシステム | 「なぜこれを変えるべきか」 |
| 戦略的判断 | 「この前提が間違っていたらどうなるか」 |
| デザイン | 「最も厳しいユーザーの視点で見たら何が問題か」 |
| コード | 「6ヶ月後に最も後悔する設計判断はどれか」 |
| 事業計画 | 「競合が最適な対抗手を打ってきたらどうなるか」 |
| 組織変更 | 「最も声の大きくない関係者が失うものは何か」 |

焦点は Phase 3 の DA ペルソナ定義に組み込む(例:
「この [対象] について、6ヶ月後に最も後悔する設計判断を徹底的に洗い出せ」)。

## Part 4: パネルサイズのガイドライン

DA は専門家枠とは別枠で常に1名。

| 議題の性質 | 直接関連 | 受け手 | 隣接ドメイン | DA | 合計 |
|---|---|---|---|---|---|
| 単一ドメインの明確な問題 | 2名 | 0-1名 | 0名 | 1名 | 3-4名 |
| 複数ドメインが絡む問題 | 2-3名 | 1名 | 0-1名 | 1名 | 5-6名 |
| 多くのステークホルダーに影響 | 3-4名 | 1名 | 1-2名 | 1名 | 7-8名 |

- **最小**: 3名 + DA = 4名
- **最大**: 7名 + DA = 8名
- 8名を超えるとトークンコストに見合わない

## Part 5: ドメイン別アーキタイプ早見表

以下は**一例**。そのままコピーするものではなく、
原理1-5 に基づいて議題ごとに動的に判断する。

### Web サイト・UI

| 専門家 | 主な観点 | 見逃しやすい盲点 |
|---|---|---|
| フロントエンドエンジニア | 実装可能性、パフォーマンス、レスポンシブ | 技術的に可能だが UX が悪いパターン |
| UI/UX デザイナー | ユーザー体験、情報設計、インタラクション | 美しいが使いにくいデザイン |
| アクセシビリティスペシャリスト | WCAG 準拠、スクリーンリーダー、色覚多様性 | 健常者視点だけで設計された UI |
| コンテンツストラテジスト | メッセージング、トーン、情報階層 | UI は良いがコンテンツが伝わらない |
| SEO/マーケティング | 検索性、コンバージョン、ユーザー獲得 | 美しいが見つけてもらえないサイト |

### モバイルアプリ

| 専門家 | 主な観点 |
|---|---|
| モバイルエンジニア | プラットフォーム固有の制約、パフォーマンス |
| インタラクションデザイナー | ジェスチャー、ナビゲーション、片手操作 |
| プロダクトマネージャー | ユーザー価値、優先順位、MVP 定義 |
| QA エンジニア | エッジケース、デバイス互換性 |

### コード・アーキテクチャ

| 専門家 | 主な観点 |
|---|---|
| ソフトウェアアーキテクト | 設計パターン、拡張性、保守性 |
| セキュリティエンジニア | 脆弱性、認証・認可、データ保護 |
| パフォーマンスエンジニア | レイテンシ、スループット、スケーラビリティ |
| DevOps/SRE | 運用性、監視、デプロイ、障害対応 |
| テストエンジニア | テスタビリティ、カバレッジ戦略 |

### 事業戦略

| 専門家 | 主な観点 |
|---|---|
| 戦略コンサルタント | 市場分析、競合ポジショニング、参入障壁 |
| ファイナンシャルアナリスト | 収益性、コスト構造、投資対効果 |
| オペレーションマネージャー | 実行可能性、スケーラビリティ、リソース制約 |
| ユーザーリサーチャー | 顧客ニーズ、行動データ、市場のペイン |
| 法務・コンプライアンス | 規制リスク、契約、知的財産 |

### デザイン・ブランディング

| 専門家 | 主な観点 |
|---|---|
| ブランドストラテジスト | 一貫性、ポジショニング、差別化 |
| ビジュアルデザイナー | 色彩、タイポグラフィ、構成、余白 |
| コピーライター | メッセージング、トーン、ストーリー |
| ユーザーリサーチャー | ユーザー感情、第一印象、信頼感 |

### 組織・チーム

| 専門家 | 主な観点 |
|---|---|
| 組織設計コンサルタント | 構造、報告ライン、意思決定プロセス |
| HR ストラテジスト | 採用、育成、リテンション、文化 |
| アジャイルコーチ | プロセス、チーム自律性、継続改善 |
| 行動科学者 | モチベーション、バーンアウト、チームダイナミクス |

### 教育・学習

| 専門家 | 主な観点 |
|---|---|
| インストラクショナルデザイナー | 学習設計、到達目標、評価 |
| 認知科学者 | 記憶、認知負荷、学習理論 |
| UX デザイナー | 学習体験、エンゲージメント |
| 教育者/講師 | 現場の実践知、学習者の実態 |

### コンテンツ・メディア

| 専門家 | 主な観点 |
|---|---|
| 編集者 | 構成、読みやすさ、事実確認 |
| ライター | 文体、トーン、ストーリーテリング |
| SEO スペシャリスト | 検索性、キーワード戦略 |
| ソーシャルメディアストラテジスト | 拡散性、プラットフォーム特性 |

### AI/ML システム

|---|---|

| MLOps/AI プラットフォーム | デプロイ、監視、ドリフト検出 |
| AI 倫理専門家 | バイアス、説明可能性、公平性 |
| プロダクトマネージャー | ユースケース適合性、ユーザー受容性 |
