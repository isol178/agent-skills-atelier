# Brag Refiner — 事実→インパクト変換インタビュー

## このスキルの目的

多くの人は「やったこと」は書けるが「それがどう価値を生んだか」への変換が苦手。
このスキルは、ユーザーが書いた事実ベースのログに対してインタビュー（質問）を行い、
評価面談で使えるインパクト記述に変換する。

## 2つのモード

このスキルは利用タイミングに応じて動作を調整する。

### 週次モード（軽量）
- トリガー: 「今週やったこと」「週次ログ」「今週の振り返り」など、1週間分のログが対象のとき
- インタビューは1〜2問に絞る。記憶が新鮮なので深掘りしなくても情報が出やすい。
- 出力は簡潔に。各エントリ1〜2文のインパクト記述 + 数字があれば添える程度。
- カテゴリ分類は不要。時系列のまま洗練する。
- 所要時間の目安: 1往復のインタビュー → 出力、計2〜3ターンで完了。

### 期末モード（フル）
- トリガー: 「期末」「自己アピール」「評価面談」「半期分」など、長期間のまとめが対象のとき
- インタビューは複数ラウンド（必要に応じて）。半年前の記憶は薄れているので丁寧に引き出す。
- 出力はカテゴリ別に構造化し、インパクト順にソートする。
- 所要時間の目安: 2〜4往復のインタビュー → 出力 → レビュー。

### モードの判定
素材の量と文脈から自動判定する。1週間分程度なら週次モード、複数週〜半期分なら期末モード。
判断に迷ったら「週次の振り返りですか、それとも期末のまとめですか？」と聞く。

## ワークフロー

### Phase 1: 素材の受け取りと分類

1. ユーザーからログ/メモを受け取る（テキスト貼り付け or ファイル）
2. モードを判定する（週次 or 期末）
3. 各エントリを以下の観点でスキャンし、**不足情報を特定**する:
   - **What（何をしたか）**: 通常これは書いてある
   - **Why（なぜやったか）**: 課題・背景・動機
   - **Impact（何が変わったか）**: 定量的な効果、定性的な変化
   - **Scope（影響範囲）**: 自分だけ / チーム / 組織 / 顧客
   - **Difficulty（難しさ）**: 技術的チャレンジ、制約、トレードオフ

### Phase 2: インタビュー

不足情報を埋めるための質問をする。以下のルールに従う:

- **週次モードでは質問は最大2つ**。サクッと終わらせる。
- **期末モードでは質問は最大5つ**。多すぎると負担になるのでこれが上限。
- 優先度が高い項目（Impact, Why）から聞く。
- 項目ごとにまとめて聞く（「○○について3点聞かせてください」ではなく「○○の件：背景は？」「○○の件：数字はある？」と個別に）。
- ユーザーが「わからない」「覚えてない」と言ったら深追いしない。推測で補える部分は提案する。
- 全エントリについて聞く必要はない。明らかにインパクトが大きいもの、情報が足りないものに絞る。

**質問テンプレート（参考）:**

| 不足情報 | 質問例 |
|---------|--------|
| Why | 「○○に取り組んだきっかけ・背景は何でしたか？ 誰かから依頼された？ 課題があった？」 |
| Impact（定量） | 「○○の結果、数字で測れる変化はありましたか？ 処理時間、エラー率、コスト、工数など」 |
| Impact（定性） | 「○○の後、チームや顧客の反応はどうでしたか？ 何か変わりましたか？」 |
| Scope | 「○○の恩恵を受けたのは誰ですか？ 自分のチームだけ？ 他部署やクライアントにも？」 |
| Difficulty | 「○○で特に難しかったこと、工夫したことはありますか？」 |

### Phase 3: インパクト記述の生成

インタビュー結果を踏まえて、以下の形式でまとめる。

#### 出力フォーマット

**週次モード:**

```markdown
## YYYY-Www (M/D–M/D)

### 成果
- [プロジェクト名] インパクト記述（1〜2文。数字があれば添える）
- [プロジェクト名] インパクト記述

### 学び・判断
- 技術的な発見や設計判断があれば（なければ省略可）
```

週次モードでは、ユーザーの既存bragファイルにそのまま追記できる形式で出力する。

**期末モード:**

```markdown
# 自己アピール [期間]

## [カテゴリ名]（例：技術的成果、プロジェクト推進、組織貢献）

### [成果タイトル]
[1〜3文のインパクト記述]

- **背景**: [課題・動機を1文で]
- **アクション**: [自分が具体的にやったことを1〜2文で]
- **成果**: [定量的・定性的な結果]
- **影響範囲**: [誰が恩恵を受けたか]
```

#### 記述の原則

- **主語は「私」にする**: 「チームで対応した」ではなく「私が○○を主導し、チームと連携して対応した」
- **動詞を強くする**: 「参加した」「関わった」→「設計した」「提案して導入した」「主導した」
- **数字を入れる**: 「高速化した」→「処理時間を20分→1分に短縮（20x改善）」
- **Before/Afterを明確にする**: 変化が伝わるように
- **盛らない**: 事実に基づく。ユーザーが言っていないことを勝手に追加しない。ただし、ユーザーが過小評価している成果は「これはもっと強く書いていいと思います」と提案する。

### Phase 4: レビューと調整

出力を見せて確認を取る。以下を聞く:

- 「事実と違うところはありますか？」
- 「書きぶりが強すぎる/弱すぎるところはありますか？」
- 「追加したい成果はありますか？」

修正があれば反映して最終版を出力する。

## 注意事項

- ユーザーの会社の評価制度やフォーマットが指定されている場合はそれに合わせる。
- 日本語で書かれたログは日本語で出力する。英語なら英語。
- 最終出力はMarkdownファイルとして保存し、ユーザーに渡す。
- インタビューの途中でユーザーが「もういい、今ある情報でまとめて」と言ったら、手持ちの情報で最善を尽くす。完璧を求めてプロセスを引き延ばさない。
