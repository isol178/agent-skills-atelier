# Scenario Design

評価シナリオの選び方と設計パターン。SKILL.md の「1. ベースライン準備」で参照される。

## 基本構成

**2 〜 3 種のシナリオを用意する。1 本では過適合、4 本以上はコスト過大。**

推奨構成:
- 中央値 1 本: 対象プロンプトが「よくある状況」で使われるケース
- Edge 1 〜 2 本: プロンプトの曖昧性・境界条件を突くケース
- （オプション）hold-out 1 本: 収束判定時の過適合チェック用。他と分けて保管

## 中央値シナリオ

「この skill / プロンプトを使うユーザーが、最も頻繁にぶつかる状況」を 1 つ選ぶ。

良い中央値シナリオ:
- 対象プロンプトの main use case がど真ん中で当たる
- 要件チェックリストが自然に 3 〜 5 項目書ける
- 実行者が迷わず着手できる（タスクの意図が明確）

悪い中央値シナリオ:
- 対象プロンプトと直交する部分まで判定対象に含めている
- タスクが大きすぎて 30 分以上かかる（subagent コストが過大）
- 正解が 1 通りしかない＝プロンプトの曖昧性を測れない

## Edge シナリオ

対象プロンプトの境界条件・曖昧性を突く。empirical tuning の **本命はここ**。精度が中央値で 100% でも、edge で落ちるなら改善の余地がある。

### Edge の選び方

以下のいずれかを狙う:

1. **指示の解釈が割れそうな状況**: 「複数あるときどうする？」「指示の優先順位がぶつかるとき？」
2. **エッジ条件**: 空入力 / 極大入力 / 不整合な入力
3. **類似 skill / プロンプトとの境界**: 「この skill じゃなく別の skill で処理すべき？」の判断を含む
4. **ドメインシフト**: 主要用途から少し外れたがカバーすべき使い方

### Edge シナリオの例

対象が「Markdown を清書する skill」なら:
- 中央値: きれいな Markdown を受け取って体裁を整える
- Edge 1: 見出しレベルが壊れた Markdown（判断の余地あり）
- Edge 2: HTML タグが混在した Markdown（skill の適用範囲境界）

## シナリオと要件チェックリストの紐付け

各シナリオに固有の要件チェックリストを書く。使い回さない。

理由: シナリオごとに「成果物に求められるもの」は違う。中央値用のリストを edge にそのまま適用すると、edge 固有の判断（例: 「不整合なときの挙動選択」）が評価対象から落ちる。

### 最小構成の目安

| シナリオ種別 | 項目数目安 | うち [critical] |
|---|---|---|
| 中央値 | 3 〜 5 | 1 〜 2 |
| Edge | 4 〜 7（分岐条件を含めるため増える） | 1 〜 2 |

## Hold-out シナリオの扱い

**推奨: baseline 設計と同時に用意する**。後出しで作ると、iter を重ねて浮き彫りになった対象プロンプトの弱点を知った状態で設計することになり、意図せず難しめ / 簡単めに振れやすい。

hold-out の要件:
- 中央値とも edge とも重複しない状況
- 難易度は中央値と edge の中間程度
- 要件チェックリストの項目数・[critical] 数は中央値・edge と同等の粒度で揃える

後出しで作る場合は、作成時に iter 1 時点の評価記録を見ずに書く（ノイズが入る）。

## アンチパターン

- **シナリオが抽象的すぎる**: 「ユーザーが何か質問する」では判定が書けない
- **中央値が 2 本ある**: どちらかを edge に振り替えるか捨てる。同質シナリオは情報量が増えない
- **全 edge で同じ曖昧性を突く**: 同じ不明瞭点しか検出できない。種類を散らす
- **シナリオ修正を iter の途中で行う**: 前後比較が崩れる。変えるなら iter 1 に戻す
