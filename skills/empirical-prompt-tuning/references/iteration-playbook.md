# Iteration Playbook

修正の波及パターンと、修正案の立て方の具体事例集。SKILL.md の「修正の波及パターン」節から参照。

## 波及パターンの 3 種

修正→効果は線形ではない。見積もりと実測のズレは次の 3 パターンで起こる。

### パターン 1: 保守的に振れる（見積もり > 実測）

**症状**: 1 修正で複数の軸改善を狙ったが、1 軸しか動かなかった。

**典型例**:
- 「この節を書き直せば精度もステップ数も所要時間も改善するはず」と見積もり → 精度のみ改善、他は変わらず
- 根本原因: 判定文言のうち 1 つにしか該当する修正になっていなかった

**教訓**: 1 修正で複数軸は狙いにくい。1 修正 1 軸改善を基本に見積もる。

### パターン 2: 上振れ（見積もり < 実測）

**症状**: 1 修正で 1 軸改善を狙ったが、複数軸が同時に改善した。

**典型例**:
- 「手順に最小完成例を inline に追加する」修正 → 精度（例示があるので正解に近づく）、ステップ数（references を読まなくなる）、duration（同上）、不明瞭点（曖昧さの自己解消）の 4 軸が同時改善
- 根本原因: 構造的な情報（コマンド + 設定 + 期待出力の組合せ）が複数の判定文言を同時に満たした

**教訓**: 「情報の組合せが同時に効く」修正は high-leverage。該当しそうな iter は投入優先度を上げる。

### パターン 3: ゼロ振れ（見積もり > 0、実測 = 0）

**症状**: 修正したのに何の軸も動かなかった。

**典型例**:
- 「明瞭性を上げるため節を細分化」→ 全軸変動なし
- 根本原因: 軸名（「明瞭性」）から推測した修正で、具体的な判定文言（例: 「要件 3: 出力に X フィールドが含まれる」）のどれにも届いていなかった

**教訓**: 軸名と判定文言は別物。**修正前に「この修正が要件チェックリストのどの項目を動かすか」を subagent に言語化させる**。紐付かないなら修正を作り直す。

## 修正案の立て方

### ステップ

1. **不明瞭点を抽出**: subagent のレポートから「不明瞭点」「裁量補完」を集める
2. **系統に分類**: 同じ根本原因に由来するものをまとめる。例:
   - 「用語 X の定義がない」系
   - 「エッジケース Y のハンドリングが未定義」系
   - 「最小完成例がない」系
3. **1 系統を選ぶ**: 同じ iter では同系統のみ扱う。複数系統は次回に回す（SKILL.md 「1 iter 1 テーマ」参照）
4. **修正案を書く**: 具体的な文言案まで書く。抽象化して丸めない
5. **判定文言に紐付ける**: 「この修正は要件 3 の × → ○ を狙う」「副次的に不明瞭点 A も解消される」のように明示
6. **適用**: プロンプト本体を編集

### 紐付けの例

修正案: 「ワークフロー節の冒頭に `.env.example` のサンプルを inline で追加する」

紐付け:
- 主: 要件 2 `[critical] 環境変数 API_KEY が設定されている` の × → ○ を狙う
- 副: 不明瞭点「環境変数ファイルはどこに置く？」の解消
- 期待される副産物: ステップ数の減少（references を読む動作が不要になるため）

このように **主 / 副 / 副産物** に分けて書くと、実測時に波及パターン（保守・上振れ・ゼロ振れ）のどれだったか事後評価できる。

## 典型的な修正レシピ

プロンプト改善で頻出する修正パターン:

### 1. 最小完成例 inline 追加

症状: `tool_uses` が 1 シナリオだけ異常に多い、または裁量補完が頻発

修正: 主要 use case の「入力 → 出力」サンプルを本文先頭近くに配置

効果: 精度改善 + ステップ数減 + duration 減（high-leverage）

### 2. 用語定義の追加

症状: 同じ用語で解釈が割れている（複数 subagent のレポートを突き合わせると差が見える）

修正: 用語集節を作る、または初出箇所で括弧書き定義

効果: 不明瞭点の解消。精度への影響は間接的

### 3. 境界条件の明示

症状: edge シナリオで特定判断が割れる（裁量補完）

修正: 「X のときは Y とする」形式の条件分岐を追加

効果: edge シナリオの精度改善。中央値には影響しないことが多い

### 4. NG 例（Red flag）追加

症状: 実行者が毎回同じ合理化をする（例: 「自己再読で済ませた」「1 シナリオで済ませた」）

修正: 「出てくる合理化 | 実態」形式の対比表を追加

効果: 不明瞭点の解消、再試行回数の減少

### 5. 手順の節番号化

症状: 実行者が手順の順序を間違える、抜かす

修正: 番号付きリスト化、各ステップに「完了条件」を添える

効果: 精度改善、再試行回数減

## 避けるべき修正

- **MUST / NEVER の乱発**: 効かないことが多い。why を書いて納得させる方が効く
- **冗長な警告の追加**: 実行者は本文から必要情報を抽出するのに時間を食う。trade-off を意識
- **判定文言に紐付かない一般論**: 「明瞭性向上」「構造改善」などは実測で動かない
- **シナリオ側の変更による精度改善**: 本末転倒
