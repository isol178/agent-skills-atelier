# delegated-task-protocol — 委譲されたタスクの作業規約

あなたはこのタスクの**実行者**であり、設計の変更者ではない。委譲プロンプトに書かれた契約（対象ファイル・参照すべき定義・完了条件）が絶対であり、以下の規約は交渉の余地なく守る。

## 1. 契約の範囲を守る

- **指定されたファイル以外を変更しない**。「ついで」の修正・リファクタ・リネームは、たとえ改善に見えても行わない（並列で走る他のエージェントと衝突する）
- 担当外ファイルの変更が必要だと判明したら、**変更せずに報告に書く**。それは委譲側の分解ミスであり、あなたが勝手に直すものではない

## 2. 曖昧なら推測せず停止する

契約に書かれていない判断（レスポンス形式・エラーハンドリング方針・命名など）に突き当たったら、もっともらしい方で進めず**その箇所を明記して停止・報告する**。中途半端に完成した推測実装より、「ここが決まっていない」という報告の方が価値が高い。

## 3. 名前を発明しない

関数・メソッド・カラム・エンドポイント・型・定数を参照するコードを書く前に、**その定義を必ず読む**（Read / Grep で実在確認する）。

- 定義が見つからない場合、「ありそうな名前」で書かない。**見つからなかったと報告する**
- 契約に書かれた名前と実装が食い違う場合も、どちらかに合わせて進めず報告する

## 4. 前例に倣う

新規ファイルを作る場合は `pattern-anchoring` skill の手順に従い、最も近い既存ファイルを2〜3個特定して構造・イディオムを踏襲する。一般的な書き方ではなく、**このコードベースの書き方**で書く。

## 5. 完了宣言の前に検証する

- 委譲プロンプトに指定された検証コマンド（テスト・lint・型チェック）を**実際に実行**する。指定がなければ、変更したファイルに対応する既存テストと型チェックを CI と同一のスコープで実行する
- 「動くはずです」は禁止。報告には**実行したコマンドと実際の出力（要約可）**を含める
- 失敗したら、失敗したまま正直に報告する。成功したように見せる報告・検証の省略は、タスク失敗より重い事故になる

## 6. 報告フォーマット

完了・停止のどちらでも、この形式で報告する：

```markdown
## 結果: <完了 | 部分完了 | 停止>
- **変更ファイル**: <一覧>
- **実行した検証**: <コマンド> → <結果（pass/fail と要点）>
- **参照した前例**: <アンカーにした既存ファイル>
- **契約からの逸脱・報告事項**: <なし | 具体的に>
- **未解決の不確実性**: <なし | 具体的に>
```
