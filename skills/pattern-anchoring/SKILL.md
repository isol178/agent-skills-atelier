# pattern-anchoring — 前例に倣って書く

新規ファイルの品質を最も安く上げる方法は、うまく書くことではなく**既にあるものに合わせる**ことである。書き始める前にアンカーを特定し、書き終わったら何に倣ったかを報告する。

## 1. アンカーを2〜3個特定する

これから作るファイルと**同じ層・同じ種類**の既存ファイルを探す：

- 置き場所の兄弟を見る：`ls` で同じディレクトリ・並列ディレクトリの構成を確認する
- 同じ役割を grep する：新規が一覧エンドポイントなら既存の一覧エンドポイントを、フォーム画面なら既存のフォーム画面を探す
- 最も新しく・最も近いものを優先する（古いファイルは移行前の流儀が残っていることがある）

アンカーなしで書き始めない。**1個では流儀か偶然か区別できないので2〜3個**見る。

## 2. アンカーから読み取るもの

- import の書き方・順序、export のスタイル（named / default、barrel の有無）
- 命名規則（ファイル名・関数名・テスト名）
- エラーハンドリングとバリデーションの置き場所
- レスポンス・戻り値の形式（エンベロープの有無、キー名）
- テストなら：fixture・モックの流儀、setup / teardown のパターン
- コメント・docstring の密度（多くも少なくもしない）

2〜3個のアンカーで**共通している部分が「流儀」**であり、踏襲対象。1個にしかない部分は個別事情の可能性がある。

## 3. 逸脱の扱い

- タスクの要件がアンカーの構造と合わない場合のみ逸脱してよい。その場合、**逸脱した点と理由を報告に明記する**（黙って新しいパターンを持ち込まない）
- アンカー間で流儀が割れている場合は、新しい方に合わせ、割れていたことを報告する

## 4. アンカーが見つからない場合

このコードベースに前例のない種類のファイルなら、発明せずに**「前例なし」と報告した上で**、使用フレームワークの最も標準的なパターン（公式ドキュメントの基本形）で書く。独自の抽象化・独自のディレクトリ構成をこの局面で導入しない。

## 5. 報告

完了報告に「参照した前例」としてアンカーにしたファイルパスを列挙する。これによりレビュー側は「何に倣ったか」を検証でき、間違ったアンカー選択（例：移行前の古い実装に倣った）を早期に発見できる。
