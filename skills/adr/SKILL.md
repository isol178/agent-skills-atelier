# ADR (Architecture Decision Record) 起票

プロジェクトにおける非自明な技術的決定を `docs/decisions/NNNN-slug.md` に記録し、将来の参照と文脈の保存を確実にする。

## このスキルの使用/非使用

**使う**:
- 新しいライブラリやフレームワークの導入
- システムの主要なコンポーネントの分割・統合方針
- 永続化層（DB）のスキーマ設計方針や選定
- セキュリティ、パフォーマンス、可用性に影響を与える重要なトレードオフの決定
- 既存の ADR を覆す、または修正する決定

**使わない**:
- 些末なリファクタリング
- 明確なバグ修正
- コーディング規約の微調整（別途 Linter や Wiki で管理すべきもの）
- 決定が未定で、単なる議論のメモを残したいとき

## 全体フロー

1. **Intake & Research**: 決定の背景、代替案、トレードオフの情報を収集
2. **Next Number Check**: 次に使用する ADR 番号（NNNN）を特定
3. **Drafting**: テンプレートに基づき ADR ファイルを作成
4. **Supersede Check**: 既存の ADR との整合性を確認し、必要なら更新
5. **Final Review**: ユーザーに内容を確認し、必要なら修正

---

## 手順詳細

### 1. 情報の収集 (Intake & Research)

決定を記述するために以下の情報を codebase や対話履歴から収集する。
- 決定が必要になった背景（具体的な課題、ビジネス要件）
- 検討した代替案（少なくとも1つ以上）
- 選定した理由と、あえて選ばなかった案の却下理由
- この決定によって生じる正負の影響（トレードオフ）

### 2. 次の番号の特定 (Next Number Check)

```bash
mkdir -p docs/decisions
ls docs/decisions/ | grep -E '^[0-9]{4}' | sort | tail -1
```
※ ファイルがない場合は `0001` から開始する。

### 3. ADR ファイルの作成 (Drafting)

`docs/decisions/NNNN-<slug>.md` を以下のテンプレートで作成する。
- **<slug>**: タイトルを簡潔な英小文字とハイフンで繋げたもの（例: `use-tailwind-css`）。
- **ステータス**: 
    - 通常は `Accepted`（決定済み）とする。
    - 複数の案からユーザーに選んでもらう段階では `Proposed` とする。

```markdown
# NNNN. <タイトル>

- **日付**: YYYY-MM-DD
- **ステータス**: Accepted
- **決定者**: [Agent Name] (via Gemini CLI / Claude Code)

## コンテキスト

なぜこの判断が必要になったか。背景、制約、前提条件を記述。
「XXXという課題があり、YYYという制約の中で解決する必要があった」という形式が望ましい。

## 決定

何を選んだか。一文で明確に。
「ZZZを採用する」

## 理由

なぜその選択をしたか。
- 代替案 A と比較して XXX の点で優れているため
- プロジェクトの長期的なメンテナンス性を優先したため、など

## 代替案

| 案 | メリット | 却下理由 / デメリット |
|---|---|---|
| ... | ... | ... |

## 結果

この決定がもたらすトレードオフ・今後の影響。
- ポジティブな影響: [例: 開発速度の向上]
- ネガティブな影響: [例: 新しい依存関係の導入による学習コスト]

## 参考

- 関連する spec / issue / PR
- 関連する既存 ADR（相対パスでリンク: `[NNNN](NNNN-slug.md)`）
```

### 4. 既存 ADR との整合 (Supersede Check)

新しい判断が既存の ADR (例: `0005-old-slug.md`) を覆す場合は以下を行う。

1. **新 ADR**: `- **Supersedes**: [0005](0005-old-slug.md)` という行をステータスの下に追加する。
2. **旧 ADR**: ステータス行を `Superseded by [NNNN](NNNN-new-slug.md)` (新番号と新ファイルへのリンク) に書き換える。

**旧 ADR の更新例**:
```markdown
- **ステータス**: Superseded by [0008](0008-new-slug.md)
```
※ 本文（過去の決定内容）は歴史的経緯として残すため、書き換えない。

### 5. ユーザーへの提示と完了

作成した ADR の内容をユーザーに提示し、承認を得る。
また、ADR は関連するコード変更と同じコミット/PRに含めることを推奨する。

## 高品質な ADR を書くためのヒント

- **客観性**: 「〜だと思う」ではなく「〜という理由で〜を選択した」と事実と論理に基づいて書く。
- **検索性**: タイトルと Slug には検索されやすいキーワードを含める。
- **トレードオフ**: 完璧な解決策はないことを前提に、何を諦めて何を得たかを正直に書く。これが数年後の開発者にとって最も価値のある情報になる。
- **不可逆性**: あとで簡単に変えられること（Type 2 Decision）よりも、一度決めると変えるのが難しいこと（Type 1 Decision）を優先的に ADR に残す。

## 禁止事項

- 既存の ADR ファイルを（ステータス更新以外で）削除または大幅に内容変更すること。
- ADR 番号を重複させること。
- 背景（Context）を省略すること。決定事項だけならコードを見ればわかるため。
