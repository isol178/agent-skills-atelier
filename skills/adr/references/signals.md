# 起票シグナル詳細

ADR を起票すべきかの判断において、`SKILL.md` の判定軸(why 復元テスト)を補助する6つのシグナル。1つでも該当すれば起票候補となる。

各シグナルには「why が消失するとどう壊れるか」の失敗シナリオを併記している。これは抽象論ではなく具体的な恐怖で起票判断を動機づけるためである。

---

## S1. 自然なデフォルトの放棄

ライブラリ/フレームワーク/言語の "普通の使い方" を意図的に避けて別の道を選んだ場合。

### 具体例

- パスワード認証 → IAM 認証への移行(DB 接続の根本変更)
- ORM のデフォルト命名規約を上書き(snake_case ではなく camelCase など)
- フレームワーク標準のルーティングを使わずカスタム実装
- TypeScript の strict mode を一部だけ無効化
- React のキー指定を `index` ではなく特定 ID で行う(再描画挙動が変わる)

### 失敗シナリオ

なぜデフォルトを使わないのかが残らないと、**親切な開発者が "シンプル化" の名目で元に戻してしまう**。コードレビューで「これ、標準のやり方でいいのでは?」という指摘に反論する根拠が消える。

---

## S2. 非自明な技術制約への対応

ライブラリ/ランタイム/インフラ固有の罠を回避するための設計。コードを見ただけでは「なぜこんな冗長なことを?」と見える形になりがち。

### 具体例

- `connect()` がイベントループをブロックするため `connect_async()` を使う
- ある SDK のリトライ機構が壊れているため自前で wrapper を書く
- 特定リージョンでのみ発生するレートリミットへの対処
- Lambda コールドスタート対策の冗長な初期化コード
- ブラウザの特定バージョンでのみ発生するレンダリングバグへの workaround

### 失敗シナリオ

制約を知らない人が **「冗長だ」と判断して単純化し、本番で再発する**。元の Issue/バグレポートが ADR で参照されていなければ、再発時に同じ調査を繰り返す羽目になる。

---

## S3. 意図的な分離・抽象化

「なぜここだけ別扱い?」という疑問にコードが答えない構造。

### 具体例

- `FIREBASE_CREDENTIALS_FILE` のような環境分離(他の credentials は環境変数なのに?)
- 一見対称な2つのモジュールを別ライフサイクルで扱う
- 特定の責務だけ別サービス/別プロセスに切り出す
- 共通化できそうな2つのクラスをあえて別にしてある
- ある設定値だけハードコードで、他は外部設定

### 失敗シナリオ

統一感を求めるリファクタで **意図的な非対称性が壊される**。「DRY にしましょう」という善意の PR で、設計上の理由がある分離が消える。

---

## S4. トレードオフを伴う代替案からの選択

複数の合理的な選択肢があり、選ばなかった案にもメリットがあった場合。

### 具体例

- SQLite vs PostgreSQL(運用簡素 vs スケーラビリティ)
- モノリス vs マイクロサービス
- GraphQL vs REST
- Kafka vs RabbitMQ
- 自前実装 vs SaaS

### 失敗シナリオ

代替案の却下理由が残らないと **「やっぱりあっちが良かったのでは」が定期的に蒸し返される**。新メンバーが入るたびに同じ議論を繰り返す。

---

## S5. アーキテクチャ上重要な要件 (ASR) への対応

非機能要件(Architecturally Significant Requirement) — セキュリティ、可用性、性能、保守性 — を満たすために行う構造的な選択。AWS Prescriptive Guidance および Richards & Ford (2020) が ADR 必須として挙げる典型カテゴリ。

### 具体例

- マルチテナント分離方針(行レベル vs スキーマレベル vs DB レベル)
- 認証・認可の境界設計(API Gateway? アプリ層? 両方?)
- SLO 達成のためのリトライ/サーキットブレーカー方針
- データ暗号化の境界(at rest / in transit / in use)
- 監査ログの粒度と保持期間

### 失敗シナリオ

構造的な選択は影響範囲が広く、後から変更すると **広範囲の手戻りが発生する**。なぜこの境界に設計したかが残らないと、規制対応や監査時に説明できない。

---

## S6. 既存 ADR を覆す/修正する判断

過去の決定を変更する場合は必ず起票(Supersede 処理)。

### 取り扱い

- 旧 ADR は **本文を書き換えず歴史的経緯として残す**(immutability 原則)
- 旧 ADR の `status` のみ `superseded by NNNN` に書き換える
- 新 ADR の `supersedes: NNNN` で関係を明示
- `More Information` セクションでなぜ覆したかを説明

### 失敗シナリオ

旧 ADR を物理削除すると、**当時なぜそう決めたかの文脈が永遠に失われる**。「過去にこの方式を試して失敗した」という学習が組織に蓄積されない。

---

## シグナルに該当しない例

判定をブレさせないため、起票 **しない** ケースも明示する。

- 言語/フレームワークの慣用表現に従っただけ(例: React で `useState` を使う)
- 公式ドキュメント通りのデフォルト構成
- 一方向しか選択肢がない決定(例: 既存 DB が PostgreSQL なので PostgreSQL を使う)
- バグ修正、リファクタリング、命名変更
- コーディング規約レベルの統一(別途 Linter / Style Guide で管理)
- プロトタイプ段階の暫定実装(本実装時に判断する)

これらの境界で迷ったら `antipatterns.md` を参照する。
