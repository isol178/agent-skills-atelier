# ADR アンチパターン

「これは ADR ではない」の代表例と、なぜそうなのかの説明。Zimmermann "How to create ADRs — and how not to" のチェックリストに準拠。

ユーザーから「これって ADR にすべき?」と聞かれた時、または起票判断に迷った時に参照する。

---

## 議論メモ

**症状**: 結論が出ていない状態で「議論したから記録に残す」と称して書く。

**なぜダメか**: ADR は **決定済み** の判断を保存するもの。未決の議論は別の場所(Slack、ミーティングメモ、RFC ドラフト)で扱うべき。未決状態の文書を ADR として残すと、後から見た人が「これは確定事項なのか?」と混乱する。

**対処**: 結論が出てから書く。議論プロセス自体を残したい場合は ADR とは別に RFC や Design Doc として管理する。

---

## コーディング規約

**症状**: 「変数名は camelCase で書く」「インデントは 2 スペース」のような規約を ADR にする。

**なぜダメか**: コーディング規約は Linter / Formatter で機械的に強制すべきもので、判断記録ではない。ADR にすると「規約は守るべきもの」が「過去にこう決めた歴史」として相対化されてしまう。

**対処**: `.eslintrc`, `.prettierrc`, `pyproject.toml` などの設定ファイルと、CONTRIBUTING.md / Style Guide で管理する。ただし「なぜこの規約を選んだか」が非自明な場合(例: チーム内で議論があった)は ADR にしてもよい。

---

## チュートリアル・使い方ガイド

**症状**: 「Docker でローカル環境を立ち上げる方法」「デプロイ手順」を ADR にする。

**なぜダメか**: 使い方の説明は判断ではなく手順書。README.md や docs/ に書くべき内容。

**対処**: README.md / docs/ / Runbook で管理する。ただし「なぜ Docker を採用したか」「なぜこのデプロイ方式を選んだか」は ADR の対象。

---

## 愚痴・後付けの正当化

**症状**: 「このライブラリは使いにくいが仕方なく使う」「歴史的経緯でこうなっている」とだけ書く。

**なぜダメか**: 判断の根拠が示されていない。「仕方なく」では将来の開発者が「じゃあ別のに変えていいんだな」と判断する材料がない。

**対処**: 判定軸(why 復元テスト)を通る具体的な制約と判断を書く。「なぜ仕方ないのか」「他に何を試して何が問題だったか」を Context と Considered Options に展開する。書けない場合は ADR を書くタイミングではなく、まず情報収集が必要。

---

## 複数決定の混在

**症状**: 1つの ADR に「DB は PostgreSQL を使う」「キャッシュは Redis を使う」「API は GraphQL を使う」を全部詰め込む。

**なぜダメか**: 各決定が独立したライフサイクルを持つ。後で DB だけ変更したい時、ADR 全体を supersede するか部分的に書き換えるかで困る。検索時にも「PostgreSQL の件はどの ADR?」が分からなくなる。

**対処**: 1 決定 1 ADR の原則を守る。関連する決定は相互リンクで繋ぐ。複数フェーズの判断(短期/中期/長期)も別々の ADR にする。

---

## Context の欠落

**症状**: 「PostgreSQL を採用する」とだけ書いて、なぜ必要になったかが書かれていない。

**なぜダメか**: ADR の本質は **why の保全**。決定だけならコードを見ればわかる。Context が欠けた ADR は単なる結論集で、判断の再現性がない。

**対処**: 「なぜこの判断が必要になったか」「どんな課題に直面していたか」「どんな制約があったか」を冒頭に書く。inverted pyramid 文体で重要事項を先に。

---

## 代替案の省略

**症状**: 「他に選択肢はなかった」「これしか考えられなかった」として Considered Options を空にする。

**なぜダメか**: 大抵の場合、代替案は存在する。「考えなかった」のと「ない」のは違う。代替案が本当に存在しない決定は、そもそも判断ではなく強制であり、ADR の対象ではない可能性が高い。

**対処**: 最低2案を検討し、却下理由を具体的に書く。Naive な代替案("自前実装"、"何もしない")も含めると比較が立体的になる。本当に1案しかない場合は、ADR ではなく実装メモでよい。

---

## 過剰な抽象化・一般論

**症状**: 「保守性を重視する」「シンプルさが大切」のような一般論で Decision Drivers を埋める。

**なぜダメか**: 一般論は誰も反対できないので評価軸として機能しない。具体的な要件に落とし込まないと、後から「保守性とは何か」を再定義する羽目になる。

**対処**: 測定可能・観測可能な形で書く。「P95 レイテンシ 200ms 以内」「新規メンバーが1週間以内にコントリビュートできる」のような具体性。

---

## immutability の違反

**症状**: 既存 ADR の本文を後から書き換えて「より良く」する。

**なぜダメか**: ADR は時点での判断記録。後から書き換えると、当時何を考えていたかが永遠に失われる。新しい情報や判断は新しい ADR (supersede) として残すべき。

**対処**: 既存 ADR は `status` のみ変更可能(`accepted` → `deprecated` や `superseded by NNNN`)。本文の更新が必要な場合は新しい ADR を起こして supersede する。

---

## confidence の偽装

**症状**: 自信がない判断を `Confidence: high` と書く。または confidence を書かない。

**なぜダメか**: 後で再評価が必要な判断と、確信のある判断が区別できなくなる。低信頼の決定がそうと記録されていれば、状況変化時に真っ先に見直せる。

**対処**: 正直に `low` と書き、再評価のトリガー条件(例: 「ユーザー数が10倍になったら見直し」「新バージョンが出たら検討」)を併記する。
